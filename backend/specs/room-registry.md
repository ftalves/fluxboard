# Spec: Room Registry

The room registry is the single in-memory bookkeeper for all rooms in the process. It is the only code that owns the `Map<roomId, Room>` and the only code that mutates it.

This spec defines the registry's public surface, its id-generation guarantees, and the invariants it upholds across the whole set of rooms. The per-room phase machine (creation, join, disconnect, destroy) is in [`room-lifecycle.md`](room-lifecycle.md). Wire shapes are in [`wire-protocol.md`](wire-protocol.md).

---

## Purpose

The registry exists to:

1. Generate unique room ids.
2. Track which rooms currently exist.
3. Publish room lifecycle events to the bus (`room.created`, `room.destroyed`).
4. Provide a single, idempotent destruction path.

It is **not** responsible for:
- Connection or socket handling.
- Per-room state mutation (delegated to `Room` and `applyEvent`).
- Seed validation (done in the HTTP handler; rules in [`wire-protocol.md`](wire-protocol.md)).

---

## API

```ts
class RoomRegistry {
  constructor(opts: { bus: EventBus; gracePeriodMs: number });

  createRoom(seed: { elements: Record<string, Element>; arrows: Record<string, Arrow> }): Room;

  getRoom(id: string): Room | undefined;

  destroyRoom(id: string, reason: 'empty' | 'shutdown'): void;

  size(): number;

  forEachRoom(fn: (room: Room) => void): void;
}
```

The constructor takes the event bus (for lifecycle event publication) and the grace period (passed through to each `Room` so it can schedule its own destroy timer). No other dependencies.

The registry is constructed exactly once, in `index.ts`. There is no module-level singleton; the instance is threaded explicitly through the connection handler context.

## Room id generation

Generated by a pure helper `generateRoomId(): string` in `realtime/rooms/roomId.ts`.

| Property | Value |
|---|---|
| Length | `ROOM_ID_LENGTH` (default `8`) |
| Charset | URL-safe base62: `A–Z`, `a–z`, `0–9` (62 characters) |
| Source of randomness | `crypto.randomBytes` (Node's `node:crypto`) |
| Collision handling | The registry retries with a fresh id on collision (see *createRoom*). Maximum 5 retries; if all five collide, throw `RoomIdExhaustionError` — at default length this is a 62⁸ ≈ 2.18 × 10¹⁴ space, so retries beyond the first are effectively impossible. |

Charset notes:
- No hyphens, underscores, or punctuation. Keeps the URL `/ws/:roomId` segment unambiguous and shell-friendly.
- No mixed-case ambiguity warning: case is preserved as generated. Two rooms with ids differing only by case (`Ab12cd34` vs `aB12cd34`) are distinct. The probability of a meaningfully confusing pair appearing together is negligible.
- Not a UUID. UUIDs are 36 chars with hyphens; 8 chars is enough entropy for an in-memory, ephemeral registry where ids vanish on process restart.

## Behaviors

### `createRoom(seed)`

**Preconditions:** seed has already been validated by the HTTP handler. The registry trusts its input.

**Behavior:**
1. Generate a candidate id via `generateRoomId()`.
2. If the id is already in the map, retry (up to 5 times). On exhaustion, throw `RoomIdExhaustionError`.
3. Construct a `Room` with:
   - `id` = the generated id
   - `state = { elements: seed.elements, arrows: seed.arrows, processedEventIds: {} }`
   - `clients = new Map()`
   - `createdAt = Date.now()`
   - `lastEmptyAt = null`
   - `destroyTimer = null`
4. Insert into the map.
5. Publish `room.created` to the bus with:
   ```ts
   { roomId, createdAt, seedElementCount: |elements|, seedArrowCount: |arrows| }
   ```
6. Return the `Room`.

The publish happens **after** the room is in the map. This ordering is significant: a worker that handles `room.created` and immediately calls `registry.getRoom(roomId)` (e.g. in a future persistence worker) must find the room. Publishing before insertion would create a window where the worker sees a roomId that does not yet exist.

| Scenario | Result |
|---|---|
| Empty seed (`{ elements: {}, arrows: {} }`) | Accepted. Room is created with empty state. |
| Id collision on first attempt | Retried with a fresh id. Both rooms continue normally. |
| Id collision 5 times in a row | `RoomIdExhaustionError` thrown. The HTTP handler maps this to `500 internal_error`. |
| Bus publish throws | The room is already in the map; the registry catches and logs, but the room remains live. (Bus `publish` is microtask-based and only logs subscriber throws — it does not itself throw under normal conditions. This branch is defensive.) |

### `getRoom(id)`

**Behavior:** Return `map.get(id)` — `Room` if present, `undefined` if not.

This is the single lookup path used by the WS upgrade handler and by `destroyRoom`. It is `O(1)` and has no side effects.

| Scenario | Result |
|---|---|
| Id is in the map | Returns the `Room`. |
| Id is not in the map (never existed, or destroyed) | Returns `undefined`. |
| Id is the empty string or otherwise malformed | Returns `undefined`. (No exception; lookups are total.) |

### `destroyRoom(id, reason)`

**Behavior:** Idempotent. Removes the room from the map and publishes `room.destroyed`. Tolerates being called with an unknown id.

1. Look up the room. If absent, return immediately. (Idempotent.)
2. If `reason === 'empty'` and `room.clients.size > 0`, abort. (See *Grace race* in [`room-lifecycle.md`](room-lifecycle.md). A reconnect that arrived between the timer firing and this callback running has restored clients.)
3. Clear `room.destroyTimer` if set.
4. For each client handle in `room.clients`:
   - Try to send `{ type: "room_destroyed", reason }`.
   - Try to close the socket with code `1001`.
   - Both wrapped in try/catch; failures are swallowed and logged.
5. Delete the room from the map.
6. Publish `room.destroyed` to the bus with:
   ```ts
   { roomId, destroyedAt: Date.now(), reason }
   ```

`destroyRoom` MUST NOT throw under any circumstances. The function is the only path out of the registry, and a thrown exception here would orphan rooms in the map. Specifically:
- A failed `send` is swallowed.
- A failed `close` is swallowed.
- A bus publish failure is swallowed.

| Scenario | Result |
|---|---|
| Id is unknown | No-op. No bus publish. |
| `reason='empty'` but a client has reconnected | Aborted at step 2. Room remains. No bus publish. |
| `reason='shutdown'` with clients still connected | Proceeds. All clients are notified and closed. |
| Multiple `destroyRoom` calls race for the same id | First call wins (deletes the entry); subsequent calls find nothing and no-op. |

### `size()` and `forEachRoom(fn)`

Read-only inspection. `size()` returns `map.size`. `forEachRoom(fn)` iterates `map.values()` and invokes `fn` on each room.

These exist for two callers:
- The shutdown sweep in `index.ts`: iterates and calls `destroyRoom(id, 'shutdown')` for each.
- An optional periodic sanity log: "registry has N rooms, oldest empty since T."

The iteration order is insertion order (standard `Map` behavior). Callers MUST NOT mutate the registry during iteration. The shutdown sweep handles this by collecting ids first, then calling `destroyRoom` on each.

## Bus integration

The registry publishes exactly two topics:

| Topic | Payload | When |
|---|---|---|
| `room.created` | `{ roomId, createdAt, seedElementCount, seedArrowCount }` | After the room is inserted into the map, in `createRoom`. |
| `room.destroyed` | `{ roomId, destroyedAt, reason }` | After the room is removed from the map, in `destroyRoom`. |

Ordering rule: `room.created` is always published before any `domain.event` for that room. `room.destroyed` is always published after the last `domain.event` for that room. This holds because the realtime layer cannot publish a `domain.event` without first looking up the room via `getRoom`, and `getRoom` returns `undefined` outside the room's lifetime.

The registry never publishes `domain.event` itself — that is the realtime layer's responsibility (see [`realtime-broadcast.md`](realtime-broadcast.md)).

## Shutdown sweep

On `SIGINT` / `SIGTERM`, `index.ts` orchestrates:

1. Stop accepting new HTTP / WS connections.
2. Snapshot the registry's room ids (`const ids = []; registry.forEachRoom(r => ids.push(r.id));`).
3. For each id, call `registry.destroyRoom(id, 'shutdown')`.
4. Tear down workers and the bus.

The snapshot in step 2 is required: `destroyRoom` mutates the map, and `forEachRoom` callers MUST NOT mutate during iteration.

After the sweep, `registry.size() === 0`. The shutdown handler does not assert this; if a leak occurred, the process is exiting anyway and the leak is logged earlier (in `destroyRoom`'s catch blocks).

## Invariants

- **Single source of truth.** The registry's internal `Map` is the only collection of rooms in the process. There is no shadow set in the bus, the workers, or anywhere else.
- **No orphans.** Every entry in the map has had `room.created` published. Every removed entry has had `room.destroyed` published (except for the no-op branches of `destroyRoom`, which had nothing to publish).
- **Lifecycle event ordering.** For a given `roomId`, `room.created` is published exactly once before any `domain.event` for that room, and `room.destroyed` is published exactly once after the last `domain.event` for that room.
- **`destroyRoom` is total and never throws.** Unknown ids are no-ops; failed socket operations are swallowed.
- **`createRoom` and `destroyRoom` are the only mutators.** `getRoom`, `size`, and `forEachRoom` are read-only.
- **Iteration order is stable** (insertion order, per `Map`). Callers must not mutate during iteration.
- **Single instance.** Constructed once in `index.ts` and threaded explicitly. No module-level singleton or globally-cached registry.

## Out of Scope (MVP)

- Persistence of the room set across process restarts.
- Eviction policies beyond the per-room grace period (LRU, max-rooms cap, etc.).
- Multi-node sharding or distributed registries.
- Room metadata (titles, owners, ACLs).
- Lookups by anything other than `roomId` (e.g. by user, by creation time).
- Custom id formats (UUIDs, slugs from creator, etc.).
- Listing endpoints (`GET /rooms`) — the registry is internal; rooms are discoverable only via the URL handed out at creation.
